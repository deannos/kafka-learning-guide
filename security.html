Production Security
// SECURITY

Kafka Security

Encryption, authentication, and authorization — everything needed to run Kafka securely in production. Never expose Kafka without these layers.

// OVERVIEW

The Three Security Layers

Encryption (TLS)
Encrypt all data in transit between clients and brokers, and between brokers. Prevents eavesdropping and man-in-the-middle attacks. Mandatory for any production deployment.
TLS 1.2/1.3mTLS
Authentication (SASL)
Verify the identity of producers, consumers, and brokers. Supports PLAIN, SCRAM-SHA-256/512, Kerberos (GSSAPI), and OAuth 2.0 (OAUTHBEARER).
SASLKerberosOAuth
Authorization (ACLs)
Control what authenticated principals can do — read, write, create, delete. Fine-grained control at the topic, consumer group, and cluster level.
ACLsRBAC
Default Kafka is Completely Open
Out of the box, Kafka has no authentication, no encryption, and no authorization. Any client that can reach the network port can read and write any topic. Never deploy to a production network without all three security layers enabled.
// ENCRYPTION

TLS Configuration

TLS (formerly SSL) encrypts the Kafka protocol in transit. You need to generate certificates for each broker and configure clients to trust them.

1
Generate CA and Broker Certificates
Create a Certificate Authority (CA), then generate and sign certificates for each broker. In production, use your organization's PKI or a tool like cert-manager on Kubernetes.
2
Configure Broker Keystores & Truststores
Import broker certs into Java keystores. Import CA cert into truststores. Both broker and client need these configured correctly.
3
Enable SSL Listener on Broker
Add an SSL listener alongside PLAINTEXT (for internal traffic). Configure clients to use the SSL listener.
4
Configure Clients
Point clients to the SSL bootstrap server and provide truststore (and keystore for mTLS). Test connectivity.
bash — Generate certificates with keytool 
# 1. Create CA key + certificate
openssl req -new -x509 -keyout ca-key -out ca-cert \
  -days 365 -subj "/CN=kafka-ca"

# 2. Create broker keystore
keytool -genkey -keystore kafka.broker1.keystore.jks \
  -alias broker1 -validity 365 \
  -keyalg RSA -keysize 2048 \
  -dname "CN=broker1.kafka.internal,OU=Kafka,O=Company"

# 3. Create CSR (Certificate Signing Request)
keytool -keystore kafka.broker1.keystore.jks \
  -alias broker1 -certreq -file cert-file

# 4. Sign CSR with CA
openssl x509 -req -CA ca-cert -CAkey ca-key \
  -in cert-file -out cert-signed \
  -days 365 -CAcreateserial

# 5. Import CA + signed cert into keystore
keytool -keystore kafka.broker1.keystore.jks -alias CARoot -import -file ca-cert
keytool -keystore kafka.broker1.keystore.jks -alias broker1 -import -file cert-signed

# 6. Create client truststore (trust the CA)
keytool -keystore kafka.client.truststore.jks \
  -alias CARoot -import -file ca-cert
server.properties — Broker TLS config 
# Listeners: keep PLAINTEXT for inter-broker, add SSL for clients
listeners=PLAINTEXT://0.0.0.0:9092,SSL://0.0.0.0:9093
advertised.listeners=PLAINTEXT://broker1:9092,SSL://broker1.kafka.internal:9093
inter.broker.listener.name=PLAINTEXT

# SSL settings
ssl.keystore.location=/etc/kafka/certs/kafka.broker1.keystore.jks
ssl.keystore.password=changeit
ssl.key.password=changeit
ssl.truststore.location=/etc/kafka/certs/kafka.broker1.truststore.jks
ssl.truststore.password=changeit

# TLS protocol versions
ssl.protocol=TLSv1.3
ssl.enabled.protocols=TLSv1.3,TLSv1.2
ssl.cipher.suites=TLS_AES_256_GCM_SHA384,TLS_CHACHA20_POLY1305_SHA256

# Require client certificates (mTLS) - optional
ssl.client.auth=required
Python — Client TLS configuration 
from kafka import KafkaProducer

producer = KafkaProducer(
    # Use the SSL listener port
    bootstrap_servers='broker1.kafka.internal:9093',
    
    # TLS configuration
    security_protocol='SSL',
    ssl_cafile='/etc/kafka/certs/ca-cert.pem',
    
    # Only needed for mTLS (client certificates)
    ssl_certfile='/etc/kafka/certs/client-cert.pem',
    ssl_keyfile='/etc/kafka/certs/client-key.pem',
    
    # Verify broker hostname in cert (always True in prod)
    ssl_check_hostname=True,
)
// AUTHENTICATION

SASL Authentication

MechanismSecurityComplexityBest For
SASL/PLAIN MEDIUM Low Dev/test. Use ONLY over TLS. Credentials in config.
SASL/SCRAM-SHA-256 HIGH Low-Medium Production default. Salted challenge-response, no plaintext password.
SASL/SCRAM-SHA-512 HIGH Low-Medium Higher security than SHA-256. Recommended for sensitive data.
SASL/GSSAPI (Kerberos) HIGHEST High Enterprise with existing Kerberos infrastructure.
SASL/OAUTHBEARER HIGH Medium Modern cloud-native. Integrates with Okta, Keycloak, Azure AD.
bash — SCRAM-SHA-512: create users on broker 
# Create user credentials (stored in ZK/KRaft)
kafka-configs.sh --bootstrap-server localhost:9092 \
  --alter --add-config \
  'SCRAM-SHA-512=[password=supersecret123]' \
  --entity-type users \
  --entity-name payments-service

# Verify user exists
kafka-configs.sh --bootstrap-server localhost:9092 \
  --describe --entity-type users \
  --entity-name payments-service
server.properties — SASL_SSL broker config 
# Use SASL_SSL (both auth + encryption)
listeners=SASL_SSL://0.0.0.0:9093
advertised.listeners=SASL_SSL://broker1:9093
security.inter.broker.protocol=SASL_SSL

sasl.mechanism.inter.broker.protocol=SCRAM-SHA-512
sasl.enabled.mechanisms=SCRAM-SHA-512

# Broker's own credentials for inter-broker auth
listener.name.sasl_ssl.scram-sha-512.sasl.jaas.config=\
  org.apache.kafka.common.security.scram.ScramLoginModule required \
  username="kafka-broker" \
  password="broker-password";

# SSL settings (same as above)
ssl.keystore.location=/certs/kafka.broker.keystore.jks
ssl.keystore.password=changeit
Python — SASL_SSL client config 
from kafka import KafkaProducer

producer = KafkaProducer(
    bootstrap_servers='broker1:9093',
    security_protocol='SASL_SSL',
    sasl_mechanism='SCRAM-SHA-512',
    sasl_plain_username='payments-service',
    sasl_plain_password='supersecret123',
    ssl_cafile='/certs/ca-cert.pem',
    ssl_check_hostname=True,
)
// AUTHORIZATION

Access Control Lists (ACLs)

ACLs define what authenticated principals can do. Enable with authorizer.class.name=org.apache.kafka.metadata.authorizer.StandardAuthorizer (KRaft) or kafka.security.authorizer.AclAuthorizer (ZooKeeper).

Principle of Least Privilege
Grant each service only the minimum permissions it needs. A payments consumer should have Read on the payments topic and its consumer group — nothing else. Never grant wildcard * permissions in production.
bash — Complete ACL setup for a microservice 
## ── Order Service (producer to orders topic) ────────
kafka-acls.sh --bootstrap-server localhost:9093 \
  --command-config /etc/kafka/admin.properties \
  --add \
  --allow-principal User:order-service \
  --operation Write \
  --operation Describe \
  --topic orders

## ── Payments Service (consumer from orders, producer to payments) ─
# Read from orders topic
kafka-acls.sh --bootstrap-server localhost:9093 \
  --command-config /etc/kafka/admin.properties \
  --add \
  --allow-principal User:payments-service \
  --operation Read --operation Describe \
  --topic orders

# Access to consumer group
kafka-acls.sh --bootstrap-server localhost:9093 \
  --command-config /etc/kafka/admin.properties \
  --add \
  --allow-principal User:payments-service \
  --operation Read \
  --group payments-cg

# Write to payments topic
kafka-acls.sh --bootstrap-server localhost:9093 \
  --command-config /etc/kafka/admin.properties \
  --add \
  --allow-principal User:payments-service \
  --operation Write --operation Describe \
  --topic payments

## ── Admin user: full cluster access ──────────────
kafka-acls.sh --bootstrap-server localhost:9093 \
  --command-config /etc/kafka/admin.properties \
  --add \
  --allow-principal User:kafka-admin \
  --operation All \
  --cluster

## ── List all ACLs ──────────────────────────────
kafka-acls.sh --bootstrap-server localhost:9093 \
  --command-config /etc/kafka/admin.properties \
  --list
OperationResourceWho Needs It
WriteTopicProducers
ReadTopic + ConsumerGroupConsumers (both needed)
CreateCluster or TopicAdmin / DevOps
DeleteTopicAdmin only
DescribeTopicAny client connecting
DescribeConfigsTopic / BrokerMonitoring tools
AlterConfigsTopic / BrokerAdmin only
AllClusterSuper users only
// CHECKLIST

Production Security Checklist

Encryption
  • TLS enabled on all listeners
  • TLS 1.2 minimum (1.3 preferred)
  • Certificates from trusted CA
  • Cert rotation process documented
  • Inter-broker traffic encrypted
Authentication
  • SASL_SSL on external listeners
  • SCRAM-SHA-512 or Kerberos
  • Unique credentials per service
  • No shared service accounts
  • Credential rotation process
Authorization
  • ACL authorizer enabled
  • Principle of least privilege
  • No wildcard (*) ACLs in prod
  • ACLs in version control (IaC)
  • Regular ACL audit
Network & Audit
  • Brokers in private subnet
  • Firewall rules restrict 9092/9093
  • Audit logging enabled
  • No public internet exposure
  • VPN or mTLS for remote access
