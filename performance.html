Performance Tuning

From 50K to 10M+ messages per second. Complete tuning guide for broker, producer, and consumer configuration.

Broker Configuration

server.properties — Production broker tuning 
## ── NETWORK THREADS ──────────────────────────────
# Threads handling incoming requests (default: 3)
num.network.threads=8

# Threads doing I/O (default: 8)
num.io.threads=16

# Send/receive socket buffer sizes
socket.send.buffer.bytes=102400
socket.receive.buffer.bytes=102400
socket.request.max.bytes=104857600

## ── LOG / STORAGE ────────────────────────────────
# Spread across multiple disk paths for parallelism
log.dirs=/data/kafka/disk1,/data/kafka/disk2,/data/kafka/disk3

# Retention
log.retention.hours=168         # 7 days
log.retention.bytes=-1           # unlimited (use hours)
log.segment.bytes=1073741824    # 1GB segments
log.segment.ms=604800000        # Roll weekly

## ── REPLICATION ──────────────────────────────────
default.replication.factor=3
min.insync.replicas=2
unclean.leader.election.enable=false  # Never sacrifice durability
auto.leader.rebalance.enable=true

## ── PERFORMANCE ──────────────────────────────────
# Max in-memory message batches
num.replica.fetchers=4
replica.fetch.max.bytes=10485760   # 10MB replica fetch

# Flush to disk (let OS manage — don't flush manually)
log.flush.interval.messages=9223372036854775807
log.flush.interval.ms=9223372036854775807

Producer Tuning

⚡
High Throughput Config
Maximize messages per second. Accept slightly higher latency in exchange for massive batching efficiency.
Throughput
⏱️
Low Latency Config
Minimize end-to-end latency. Accept lower throughput. Use for interactive applications and real-time alerts.
Latency
🛡️
High Durability Config
Zero data loss guarantee. Accept throughput trade-off. Use for financial data, audit logs, critical events.
Durability
Python — Three producer profiles compared 
# ── PROFILE 1: Maximum Throughput ─────────────────
high_throughput_config = {
    'bootstrap_servers': 'kafka:9092',
    'acks': 1,                   # Leader ack only (not all replicas)
    'linger_ms': 50,              # Wait 50ms to fill batches
    'batch_size': 131072,         # 128KB batches
    'compression_type': 'lz4',    # Fast compression
    'buffer_memory': 67108864,   # 64MB producer buffer
    'max_in_flight_requests_per_connection': 5,
}
# Result: ~800 MB/s, latency ~100ms

# ── PROFILE 2: Minimum Latency ────────────────────
low_latency_config = {
    'bootstrap_servers': 'kafka:9092',
    'acks': 1,
    'linger_ms': 0,               # Send immediately — no waiting
    'batch_size': 1,               # Effectively no batching
    'compression_type': 'none',   # Skip compression overhead
    'max_in_flight_requests_per_connection': 1,
}
# Result: ~1ms latency, lower throughput

# ── PROFILE 3: Maximum Durability ────────────────
high_durability_config = {
    'bootstrap_servers': 'kafka:9092',
    'acks': 'all',                # All ISR replicas must ack
    'retries': 2147483647,        # Retry forever
    'retry_backoff_ms': 300,
    'enable_idempotence': True,  # No duplicate messages on retry
    'max_in_flight_requests_per_connection': 5,
    'compression_type': 'snappy',
    'linger_ms': 20,
}

Consumer Tuning

Python — High-throughput consumer config 
consumer = KafkaConsumer(
    'events',
    bootstrap_servers='kafka:9092',
    group_id='analytics',
    
    # Fetch: pull large batches to maximize throughput
    fetch_min_bytes=1048576,     # Wait for 1MB before returning
    fetch_max_wait_ms=500,       # Or wait max 500ms
    max_partition_fetch_bytes=10485760,  # 10MB per partition
    max_poll_records=1000,        # 1000 records per poll
    
    # Prevent rebalance timeouts on slow processing
    max_poll_interval_ms=300000,  # 5 minutes
    session_timeout_ms=45000,
    heartbeat_interval_ms=15000,
    
    # Networking
    receive_buffer_bytes=65536,
)

# Process in parallel using thread pool
import concurrent.futures

with concurrent.futures.ThreadPoolExecutor(max_workers=8) as executor:
    while True:
        records = consumer.poll(timeout_ms=1000)
        futures = [executor.submit(process_record, r) 
                  for msgs in records.values() 
                  for r in msgs]
        concurrent.futures.wait(futures)
        consumer.commit()

// MONITORING

Consumer Lag Management

Consumer lag is the difference between the latest produced offset and the consumer's committed offset. Growing lag = consumers can't keep up with producers.

📡
Monitor Lag
Use kafka-consumer-groups.sh --describe, Kafka's JMX metrics (records-lag-max), or tools like Burrow, Lag Exporter for Prometheus/Grafana.
⚠️
Causes of Lag
Slow consumer processing, insufficient consumer instances, rebalancing storms, network bottlenecks, consumer crashes, or underpowered consumer machines.
✅
Fix Lag
Add consumers (up to partition count), increase max.poll.records, optimize processing logic, increase partitions (with caution), or use parallel processing within consumers.
bash — Monitor consumer lag 
# Show all consumer groups and their lag
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe --group order-processor

# Output:
# GROUP          TOPIC    PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# order-proc     orders   0          1000000         1000500         500
# order-proc     orders   1          998000          1000200         2200  ← HIGH
# order-proc     orders   2          1001000         1001100         100

# Reset offset (use with extreme caution in production)
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group order-processor \
  --topic orders \
  --reset-offsets --to-latest --execute
