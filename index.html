What is Apache Kafka?

Apache Kafka is an open-source distributed event streaming platform originally developed at LinkedIn in 2011 and later donated to the Apache Software Foundation. It's designed to handle trillions of events per day.

// KAFKA HIGH-LEVEL ARCHITECTURE

* = replica partition

Core Guarantees

  • Durability — Messages are persisted to disk and replicated across brokers
  • Scalability — Horizontal scaling via partitions; can handle millions of messages/sec
  • High Throughput — Sequential disk I/O, batching, and compression
  • Fault Tolerance — Replication factor ensures no single point of failure
  • Ordering — Messages within a partition are strictly ordered
💡 KEY INSIGHT
Unlike traditional message queues that delete messages after consumption, Kafka retains messages for a configurable period (default 7 days). This allows multiple consumers to independently read the same data and enables replaying events.

Core Components Deep Dive

📦 Topics & Partitions

A Topic is a named category to which records are published. Topics are split into Partitions — ordered, immutable sequences of records. Each partition is an append-only log file.

BASH — Topic Operations 
# Create a topic with 3 partitions, replication factor 2
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --topic orders \
  --partitions 3 \
  --replication-factor 2

# List all topics
kafka-topics.sh --list --bootstrap-server localhost:9092

# Describe a topic (see partition leaders, replicas)
kafka-topics.sh --describe \
  --bootstrap-server localhost:9092 \
  --topic orders

🏭 Brokers

A Broker is a Kafka server that stores data and serves client requests. A Kafka cluster consists of multiple brokers. One broker acts as the Controller (manages cluster metadata, leader elections).

📍 Offsets

Each message in a partition has a unique sequential ID called an offset. Consumers track their position using offsets — this is how Kafka enables re-reading and exactly-once processing.

Why Apache Kafka?

The Problem Before Kafka

Before event streaming, systems used point-to-point integrations. If you have 10 services that need to share data, you end up with up to 45 direct connections — a maintenance nightmare.

⚠ THE INTEGRATION HELL
N services = N×(N-1)/2 potential connections. With 100 services that's ~5,000 pipelines to maintain, each with its own failure modes, schemas, and scaling characteristics.

Real-World Use Cases

  • Activity Tracking — LinkedIn uses Kafka to track user activity (page views, clicks) at billions of events/day
  • Metrics & Monitoring — Aggregating operational data from distributed systems (think Datadog, Splunk)
  • Log Aggregation — Centralizing logs from hundreds of microservices
  • Event Sourcing — Storing state changes as an immutable event log (financial transactions)
  • Stream Processing — Real-time fraud detection, recommendations, ETL pipelines
  • Microservice Decoupling — Services communicate via events, not direct API calls

Kafka Producer API

The KafkaProducer is thread-safe and typically you create one instance per application. It handles batching, compression, retries, and serialization internally.

PYTHON — Basic Producer 
from kafka import KafkaProducer
import json

# Create producer with recommended config
producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    key_serializer=lambda k: k.encode('utf-8') if k else None,
    
    # Reliability: wait for leader + all ISR replicas to ack
    acks='all',
    
    # Retry on transient failures
    retries=5,
    retry_backoff_ms=300,
    
    # Batching for throughput
    batch_size=16384,       # 16KB batch
    linger_ms=10,            # Wait 10ms to fill batch
    
    # Compression
    compression_type='snappy',
)

# Send with key (same key → same partition = ordering)
future = producer.send(
    topic='orders',
    key='user-123',
    value={'order_id': 'ord-456', 'amount': 99.99}
)

# Block until send completes (for critical messages)
record_metadata = future.get(timeout=10)
print(f"Sent to partition {record_metadata.partition} @ offset {record_metadata.offset}")

producer.flush()  # Ensure all pending messages are sent
producer.close()

Kafka Consumer API

PYTHON — Consumer with Manual Commit 
from kafka import KafkaConsumer, TopicPartition
import json

consumer = KafkaConsumer(
    'orders',
    bootstrap_servers='localhost:9092',
    group_id='order-processor-group',
    value_deserializer=lambda m: json.loads(m.decode('utf-8')),
    
    # Manual offset control (safer than auto-commit)
    enable_auto_commit=False,
    
    # Where to start if no committed offset exists
    auto_offset_reset='earliest',  # or 'latest'
    
    # Max records per poll()
    max_poll_records=500,
)

try:
    while True:
        # Poll with 1 second timeout
        records = consumer.poll(timeout_ms=1000)
        
        for tp, messages in records.items():
            for msg in messages:
                # Process message
                process_order(msg.value)
                
        # Commit only AFTER processing (at-least-once)
        if records:
            consumer.commit()
except KeyboardInterrupt:
    pass
finally:
    consumer.close()  # Triggers graceful rebalance

Consumer Groups & Partition Assignment

A Consumer Group allows multiple consumer instances to share the load of consuming a topic. Each partition is assigned to exactly one consumer within the group at any time.

📐 PARTITION ASSIGNMENT RULE
Max parallelism = number of partitions. If you have 6 partitions and 3 consumers: 2 partitions/consumer. If you have 6 partitions and 6 consumers: 1 partition each. If you have 6 partitions and 8 consumers: 2 consumers idle. More consumers than partitions = no additional parallelism.

Rebalancing Protocols

  • Eager Rebalance (Classic) — All partitions are revoked from all consumers, then reassigned. Causes a "stop the world" pause.
  • Cooperative/Incremental Rebalance — Only the partitions being moved are revoked. Available since Kafka 2.4. Use CooperativeStickyAssignor.

JAVA — Exactly-Once (Transactional Producer) 
Properties props = new Properties();
props.put("enable.idempotence", "true");
props.put("transactional.id", "my-transactional-id");
props.put("acks", "all");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();

try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("payments", key, value));
    producer.send(new ProducerRecord<>("ledger", key, value));
    producer.commitTransaction();  // Atomic!
} catch (Exception e) {
    producer.abortTransaction();  // Rollback both
}

Kafka's Log Storage

Kafka stores data as append-only log files on disk. Each partition is a directory containing multiple segment files.

FILESYSTEM — Partition Directory Layout
/kafka-logs/orders-0/          # Topic 'orders', Partition 0
  00000000000000000000.log     # Segment: messages 0–999
  00000000000000000000.index   # Sparse offset index
  00000000000000000000.timeindex
  00000000000000001000.log     # Segment: messages 1000+
  00000000000000001000.index
  leader-epoch-checkpoint

Log Compaction

For topics used as event sourcing / change data capture, Kafka supports log compaction. Instead of time-based retention, it retains the latest value for each key, enabling topics to act as a key-value store snapshot.

💡 COMPACTION USE CASE
Database CDC (Change Data Capture). If user #123 updates their email 50 times, a compacted topic will only keep the last "user-123" record — perfect for rebuilding state from scratch without processing all historical events.

Replication & Leader Election

For each partition, Kafka elects one broker as Leader and others as Followers. All reads and writes go through the Leader.

ISR — In-Sync Replicas

The ISR is the set of replicas that are "caught up" with the leader (within replica.lag.time.max.ms). A message is only considered committed when all ISR replicas have acknowledged it (when acks=all).

ConfigRecommendedReason
replication.factor3Survive 2 broker failures
min.insync.replicas2At least 2 must ack writes
unclean.leader.electionfalsePrevent data loss (prefer unavailability)
⚠ DURABILITY FORMULA
acks=all + min.insync.replicas=2 + replication.factor=3 = zero data loss guarantee. You can lose 1 broker and still write. You can lose 2 brokers and still read.

Why Kafka Is So Fast

1. Sequential Disk I/O

Kafka only appends to the end of log files. Sequential writes on modern SSDs can exceed 600MB/s — often faster than random reads from RAM due to OS page cache optimization.

2. Zero-Copy via sendfile()

Normally: Disk → Kernel buffer → User space → Socket buffer → NIC. With zero-copy: Disk → Kernel buffer → NIC (skips user space entirely). This is the sendfile() Linux system call — Kafka uses it to send data to consumers with minimal CPU overhead.

3. OS Page Cache

Kafka doesn't maintain its own in-memory cache. It relies on the OS page cache, which is automatically warmed by recent reads. This means recently-written messages are served from RAM without any application-level caching code.

4. Batching & Compression

Messages are batched at the producer, transmitted in compressed batches, stored compressed, and only decompressed by the consumer. A snappy-compressed batch of 1000 JSON messages can be 5–10× smaller than individual messages.

ZooKeeper → KRaft

Historically, Kafka relied on Apache ZooKeeper for cluster metadata, controller election, and configuration. KRaft (Kafka Raft) — introduced in Kafka 2.8, production-ready in 3.3 — replaces ZooKeeper with a built-in Raft consensus layer.

Kafka Streams DSL

Kafka Streams is a client library (not a separate cluster) for building stream processing applications. It runs inside your application process.

JAVA — Word Count with Kafka Streams 
StreamsBuilder builder = new StreamsBuilder();

KStream<String, String> textLines = builder
    .stream("text-input");

KTable<String, Long> wordCounts = textLines
    .flatMapValues(v -> Arrays.asList(v.split("\\s+")))
    .groupBy((key, word) -> word)         // group by word
    .count(Materialized.as("word-counts")); // stateful aggregation

wordCounts
    .toStream()
    .to("word-count-output",
       Produced.with(Serdes.String(), Serdes.Long()));

KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();

KStream vs KTable

  • KStream — Unbounded stream of records. Each record is independent (think: append-only log). Example: click events.
  • KTable — Changelog stream. Each record upserts a key. Latest value per key (think: current state). Example: user profile.
  • GlobalKTable — Like KTable but replicated to all instances (good for small lookup tables).

ksqlDB — SQL on Streams

SQL — ksqlDB Streaming Queries 
-- Create a stream from a Kafka topic
CREATE STREAM orders (
    order_id VARCHAR,
    user_id  VARCHAR,
    amount   DOUBLE,
    status   VARCHAR
) WITH (
    KAFKA_TOPIC='orders',
    VALUE_FORMAT='JSON'
);

-- Push query: continuously stream results as new data arrives
SELECT user_id, SUM(amount) AS total
FROM orders
WINDOW TUMBLING (SIZE 1 HOUR)
GROUP BY user_id
EMIT CHANGES;

-- Create materialized view (persistent query)
CREATE TABLE fraud_alerts AS
    SELECT user_id, COUNT(*) AS tx_count
    FROM orders
    WINDOW TUMBLING (SIZE 5 MINUTES)
    GROUP BY user_id
    HAVING COUNT(*) > 10
    EMIT CHANGES;

Schema Registry

Schema Registry is a centralized service for managing Avro/Protobuf/JSON schemas for Kafka topics. It enables schema evolution without breaking consumers.

PYTHON — Producer with Avro Schema 
from confluent_kafka import SerializingProducer
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroSerializer

schema_str = """
{
  "type": "record",
  "name": "Order",
  "fields": [
    {"name": "order_id",  "type": "string"},
    {"name": "user_id",   "type": "string"},
    {"name": "amount",    "type": "double"},
    {"name": "timestamp", "type": "long", "logicalType": "timestamp-millis"}
  ]
}
"""

schema_registry = SchemaRegistryClient({'url': 'http://schema-registry:8081'})
avro_serializer = AvroSerializer(schema_registry, schema_str)

producer = SerializingProducer({
    'bootstrap.servers': 'localhost:9092',
    'value.serializer': avro_serializer,
})

producer.produce(topic='orders', value={
    'order_id': 'ord-001',
    'user_id': 'usr-123',
    'amount': 49.99,
    'timestamp': 1709000000000,
})

Kafka Security

Authentication (SASL Mechanisms)

  • SASL/PLAIN — Username/password. Use only over TLS. Development-friendly.
  • SASL/SCRAM-SHA-256/512 — Salted challenge-response. More secure than PLAIN.
  • SASL/GSSAPI (Kerberos) — Enterprise environments with existing Kerberos.
  • SASL/OAUTHBEARER — OAuth 2.0 token-based. Modern cloud-native setups.
  • mTLS — Mutual TLS. Certificate-based authentication.

Architectural Patterns

Outbox Pattern

Solve the dual-write problem (database + Kafka): write to an outbox table in your DB transaction, then use Debezium to capture that table and publish to Kafka. Guarantees atomicity without distributed transactions.

Event Sourcing

Store all state changes as an immutable stream of events in Kafka. Current state = replay of all events. Enables time travel debugging, audit trails, and building multiple read models from one event stream.

CQRS with Kafka

Commands go through Kafka as events. Query models (read replicas) are materialized by consumers. Decouple write throughput from read patterns.

Saga Pattern (Distributed Transactions)

Break a distributed transaction into a sequence of local transactions, each publishing events to Kafka. If a step fails, compensating transactions are published. Choreography (each service reacts to events) vs. Orchestration (coordinator topic).
