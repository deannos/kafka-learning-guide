// COMPONENTS

Core Components

01
Broker
kafka.server.KafkaServer

A broker is a Kafka server — the core process that stores data and serves client requests. A Kafka cluster is made of multiple brokers (typically 3 or more in production). One broker is elected as the Controller, which manages cluster metadata, partition leadership, and handles broker failures.

Each broker handles hundreds of thousands of reads and writes per second across thousands of partitions while using only a few percent of CPU. Brokers use the OS page cache aggressively, meaning recently-written data is served from RAM without any application-level cache management.

💡 Broker Configuration
Key broker settings: log.dirs (partition storage), num.network.threads (I/O threads), log.retention.hours (data TTL), log.segment.bytes (segment rollover size).
02
Topics & Partitions
The fundamental storage unit

A topic is a named feed of events — like a table in a database or a folder in a filesystem. Topics are split into partitions, which are the actual unit of parallelism and ordering in Kafka.

Each partition is an ordered, immutable sequence of records that is continually appended to — an append-only log file on disk. Records in a partition are each assigned a sequential identifier called the offset.

Partitions enable Kafka to scale horizontally. More partitions = higher parallelism for both producers and consumers. The total throughput of a topic scales with the number of partitions.

03
Producers
org.apache.kafka.clients.producer.KafkaProducer

A producer publishes records to Kafka topics. The producer handles partition selection — either by explicit key (same key → same partition, guaranteeing ordering per key), round-robin (default, maximizing throughput), or a custom partitioner.

Producers batch messages internally before sending. The linger.ms config controls how long to wait to accumulate a batch, and batch.size controls the max batch size. This makes Kafka's network protocol extremely efficient.

Go — kafka-go: Producer with key-based partitioning 
package main

import (
    "context"
    "encoding/json"
    "log"
    "time"
    "github.com/segmentio/kafka-go"
)

func main() {
    w := kafka.NewWriter(kafka.WriterConfig{
        Brokers:          []string{"localhost:9092"},
        Topic:            "user-events",
        Balancer:         &kafka.Hash{},                  // hash(key) % partitions
        BatchTimeout:     20 * time.Millisecond,          // wait 20ms to fill batches
        BatchSize:        64 << 10,                      // 64 KB batch size
        CompressionCodec: kafka.Snappy.Codec(),          // reduces network + storage 3–10x
        RequiredAcks:     -1,                            // acks=all: all ISR replicas must ack
    })
    defer w.Close()

    // Key ensures all events for user-123 go to same partition
    // → guaranteed ordering per user
    event, _ := json.Marshal(map[string]any{
        "type": "checkout", "cart_total": 149.99,
    })
    if err := w.WriteMessages(context.Background(),
        kafka.Message{Key: []byte("user-123"), Value: event},
    ); err != nil {
        log.Fatal("failed to write:", err)
    }
}
04
Consumers & Consumer Groups
org.apache.kafka.clients.consumer.KafkaConsumer

A consumer reads records from Kafka topics by polling for new messages. Consumers track their position using offsets — stored either in Kafka's internal __consumer_offsets topic or externally.

A consumer group is a set of consumers that cooperatively consume a topic. Kafka ensures each partition is consumed by exactly one group member at a time. This enables horizontal scaling: adding consumers to a group increases read parallelism up to the partition count.

Go — kafka-go: Consumer with manual offset commit 
package main

import (
    "context"
    "encoding/json"
    "log"
    "github.com/segmentio/kafka-go"
)

func main() {
    r := kafka.NewReader(kafka.ReaderConfig{
        Brokers:     []string{"localhost:9092"},
        GroupID:     "order-processor",
        Topic:       "orders",
        StartOffset: kafka.FirstOffset, // start from beginning if no prior offset
        MaxBytes:    10e6,              // 10 MB max fetch
    })
    defer r.Close() // triggers graceful group rebalance

    ctx := context.Background()
    for {
        // FetchMessage does NOT auto-commit — at-least-once delivery
        m, err := r.FetchMessage(ctx)
        if err != nil {
            log.Fatal(err)
        }
        var order map[string]any
        json.Unmarshal(m.Value, &order)

        // Commit only after successful processing (at-least-once)
        if err := processOrder(order); err == nil {
            r.CommitMessages(ctx, m)
        }
    }
}
05
Replication & ISR
Fault tolerance through data duplication

Each partition has one leader and N-1 followers (replicas). All reads and writes go through the leader. Followers continuously replicate the leader's data. If the leader fails, one of the in-sync followers is automatically elected as the new leader.

The ISR (In-Sync Replicas) is the set of replicas that are caught up with the leader within replica.lag.time.max.ms. A message is considered committed only when all ISR members have acknowledged it (with acks=all).

✅ Production Durability Formula
replication.factor=3 + min.insync.replicas=2 + acks=all = Zero data loss guarantee. Can tolerate 1 broker failure for writes, 2 broker failures for reads.

06
KRaft — Kafka Without ZooKeeper
Kafka Raft Metadata Mode (Production-ready: Kafka 3.3+)

Historically Kafka depended on Apache ZooKeeper for cluster coordination, controller election, and metadata storage. KRaft (Kafka Raft) replaces this with a built-in Raft consensus layer using an internal @metadata topic.

KRaft brings dramatic improvements: metadata propagation goes from seconds to milliseconds, the cluster can support millions of partitions (vs ~200k with ZooKeeper), and controller failover drops from 30–60 seconds to under a second. All new Kafka deployments should use KRaft.

Log Storage Internals

Kafka stores each partition as a directory of segment files on disk. Understanding this structure explains how Kafka achieves high performance and how retention works.

/kafka-logs/
  orders-0/                    # topic=orders, partition=0
    00000000000000000000.log   # Segment file: offsets 0–999
    00000000000000000000.index # Sparse offset index (offset→position)
    00000000000000000000.timeindex
    00000000000000001000.log   # Segment file: offsets 1000–1999
    00000000000000001000.index
    00000000000000002000.log   # Active segment (writes go here)
    00000000000000002000.index
    leader-epoch-checkpoint
  orders-1/                    # topic=orders, partition=1
    ...
🔄
Zero-Copy Transfers
Kafka uses the Linux sendfile() syscall to transfer data directly from disk to network without copying through user space. This eliminates 2 context switches and 2 data copies per message batch.
💾
OS Page Cache
Kafka doesn't manage its own memory cache. It relies on the OS page cache. Recently written data is automatically served from RAM. Kafka can consume disk at RAM speed for recent data.
📦
Compression
Batches are compressed end-to-end: stored compressed on broker, transferred compressed to consumer. Snappy/LZ4/ZSTD can reduce payload 5–10x on JSON data.
