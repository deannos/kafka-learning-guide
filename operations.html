Critical Metrics to Monitor

Kafka exposes metrics via JMX. Use Prometheus JMX Exporter + Grafana for dashboards, or Confluent Control Center for a managed UI.

Metric (JMX / Prometheus)NormalAlert ThresholdMeaning
UnderReplicatedPartitions 0 > 0 Partitions not fully replicated — broker may be down or lagging
ActiveControllerCount 1 ≠ 1 Should always be exactly 1. 0 = no controller elected
OfflinePartitionsCount 0 > 0 Partitions with no elected leader — reads/writes failing
consumer_lag (per group) Stable/decreasing Growing for >5min Consumers falling behind producers
RequestHandlerAvgIdlePercent > 30% < 20% Broker threads saturated — will cause latency spikes
NetworkProcessorAvgIdlePercent > 30% < 20% Network I/O bottleneck
BytesInPerSec / BytesOutPerSec Baseline normal Drops suddenly Traffic anomaly — producer/consumer issues
LogFlushRateAndTimeMs < 100ms p99 > 500ms Disk I/O latency impacting producers
IsrShrinkRate 0 > 0 Replicas falling out of sync — durability at risk
LeaderElectionRateAndTimeMs Rare Frequent Brokers crashing or network partitions occurring

// TROUBLESHOOTING

Common Failure Scenarios & Fixes

🔴
Consumer Lag is Growing
Consumers cannot keep up with message production rate

Symptoms: consumer_lag metric growing, consumers stuck on older offsets, processing appears slow or stalled.

Diagnosis steps:

  • Check if consumer is alive: kafka-consumer-groups.sh --describe --group <name>
  • Check consumer processing time per record — is it too slow?
  • Check if partition count limits parallelism (lag on specific partitions only)
  • Look for frequent rebalances in consumer logs (Rebalance triggered)

Fixes: Add more consumer instances (up to partition count), optimize processing logic, increase max.poll.records, increase partition count (create new topic), use parallel processing within consumer.

🔴
UnderReplicatedPartitions > 0
Partition replicas falling behind or broker down

Symptoms: Alert fires for UnderReplicatedPartitions, durability at risk, possible producer errors if min.insync.replicas not met.

Diagnosis:

  • Check if a broker is down: kafka-broker-api-versions.sh --bootstrap-server broker:9092
  • Check ISR: kafka-topics.sh --describe --topic <name> — look for ISR smaller than RF
  • Check replica lag in broker logs: search for replica.lag.time.max.ms exceeded

Fixes: If broker is down — restart it; Kafka will automatically replicate missing data. If disk I/O is too slow — tune num.replica.fetchers. If network is the bottleneck — check bandwidth between brokers.

🔴
Producer Getting NotEnoughReplicas
Can't meet min.insync.replicas requirement

Cause: Too many brokers are down. With RF=3 and min.insync.replicas=2, losing 2 brokers means producers get NotEnoughReplicasException.

Immediate fix: Bring brokers back online. Kafka will catch up replicas automatically.

Temporary workaround (data loss risk): Reduce min.insync.replicas temporarily to 1 — use only as last resort, never in financial systems.

⚠ Never do this permanently
Setting min.insync.replicas=1 means you can lose data if the single in-sync broker dies before replication catches up. It's a temporary disaster recovery step only.
🔴
Constant Consumer Group Rebalancing
Consumers keep joining and leaving the group

Symptoms: Consumer logs show repeated Revoked partitions and Assigned partitions. Processing stalls during rebalance.

Causes:

  • Consumer taking too long to process a batch — exceeds max.poll.interval.ms, broker thinks it's dead
  • Heartbeat failure — GC pauses or CPU starvation cause missed heartbeats
  • Consumer crashing and restarting repeatedly

Fixes: Increase max.poll.interval.ms, reduce max.poll.records to process faster, fix the slow processing bottleneck, switch to CooperativeStickyAssignor to minimize partition movement during rebalance.

// SCALING

Scaling Kafka

Adding Brokers

Kafka supports adding brokers to a running cluster with zero downtime. New brokers don't automatically receive existing partition replicas — you must trigger partition reassignment.

kafka-reassign-partitions.sh --generate Plan reassignment
kafka-reassign-partitions.sh --execute Start reassignment
kafka-reassign-partitions.sh --verify Check progress

Adding Partitions

You can increase partition count but never decrease it. Adding partitions to an existing topic changes key-to-partition routing for new messages — existing data in old partitions is not redistributed.

⚠ Warning
Adding partitions breaks key ordering for messages already in the topic. For ordering-sensitive topics, create a new topic with more partitions and migrate.
// MAINTENANCE

Rolling Restarts & Upgrades

Rolling restarts allow you to restart brokers one at a time without downtime. The cluster continues serving traffic through the remaining brokers.

1
Check Cluster Health First
Verify UnderReplicatedPartitions=0 and no offline partitions before starting. Never restart a broker when the cluster is already degraded.
2
Trigger Controlled Shutdown
Use kafka-server-stop.sh which triggers a controlled shutdown — Kafka migrates partition leaders away before stopping. Much faster than a hard kill.
3
Wait for Full ISR Recovery
After restarting the broker, wait until UnderReplicatedPartitions returns to 0 before proceeding to the next broker. This ensures full durability is maintained throughout.
4
Rebalance Leaders (Optional)
After all brokers are back, run preferred leader election to redistribute partition leaders evenly: kafka-leader-election.sh --all-topic-partitions --election-type preferred
// DISASTER RECOVERY

Backup & Disaster Recovery

🔁
MirrorMaker 2
Kafka's built-in cross-cluster replication tool. Replicates topics between clusters in different datacenters or cloud regions. Supports active-active and active-passive topologies. Handles offset translation so consumers can failover seamlessly.
Geo-replicationFailover
☁️
S3 Backup (Tiered Storage)
Kafka 3.6+ supports Tiered Storage — older log segments are automatically offloaded to S3/GCS/Azure Blob. Dramatically reduces broker disk requirements while retaining unlimited history.
Tiered StorageCost Saving
📸
Snapshot Backups
For KRaft clusters, take filesystem-level snapshots of the __cluster_metadata partition. For ZooKeeper clusters, back up ZooKeeper data directories. Combined with S3 tiered storage for full protection.
