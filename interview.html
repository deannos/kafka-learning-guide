// LEVEL 1

Beginner Questions

1. What is Apache Kafka and what problem does it solve?
BEGINNER ›

Apache Kafka is a distributed event streaming platform designed for high-throughput, fault-tolerant, real-time data pipelines. It solves the integration complexity problem in distributed systems.

Before Kafka, services communicated via point-to-point integrations. With N services, you need up to N×(N-1)/2 connections — each with custom schemas and failure modes. Kafka introduces a central event log that any service can write to or read from, reducing this to N connections total.

Key capabilities: publish-subscribe messaging, durable storage of events, stream processing via Kafka Streams, and high throughput (millions of messages/sec). Unlike traditional queues, messages aren't deleted after consumption — multiple consumers can read the same data independently.

2. Explain topics, partitions, and offsets.
BEGINNER›

A topic is a named feed of events — analogous to a database table or a folder. Producers write to topics; consumers read from them.

A partition is a subdivision of a topic. Each partition is an ordered, append-only log stored on disk. Partitions are the unit of parallelism — more partitions allow more producers and consumers to work simultaneously. A topic with 6 partitions can be consumed by up to 6 consumer instances in parallel.

An offset is a unique sequential identifier for each record within a partition, starting at 0. Consumers use offsets to track their position — "I've processed up to offset 4,221 in partition 0." This enables consumers to replay data, start from the beginning, or pick up exactly where they left off after a crash.

3. What is a consumer group and why is it important?
BEGINNER›

A consumer group is a set of consumer instances that cooperate to consume a topic. Kafka ensures each partition is assigned to exactly one consumer within the group at any time — this guarantees ordered processing within a partition while distributing load across the group.

Consumer groups enable two key patterns: load balancing (multiple consumers share work; adding consumers increases throughput up to partition count) and broadcast (multiple groups can all consume the same topic independently — each gets its own complete copy of all events).

The critical rule: max parallelism = number of partitions. If a group has more consumers than partitions, some consumers will be idle.

4. What is the difference between Kafka and RabbitMQ?
BEGINNER›

The fundamental difference is their model: RabbitMQ is a traditional message queue (work distribution), while Kafka is an event streaming log.

Key differences: RabbitMQ deletes messages after consumption; Kafka retains them for days/weeks and allows replay. RabbitMQ pushes messages to consumers; Kafka consumers pull at their own pace. Kafka supports millions of messages/sec; RabbitMQ tops out around 50K/sec. Kafka has native consumer groups with parallel consumption; RabbitMQ uses competing consumers with more limited semantics.

Choose RabbitMQ for task queues, complex routing, or RPC patterns. Choose Kafka for high-volume event streaming, event sourcing, CDC, or when multiple independent services need to consume the same data.

// LEVEL 2

Intermediate Questions

5. Explain Kafka's replication mechanism and what ISR means.
INTERMEDIATE›

For each partition, Kafka designates one broker as the leader and others as followers. All client reads and writes go to the leader. Followers continuously replicate data from the leader.

ISR (In-Sync Replicas) is the set of replicas that are fully caught up with the leader, within the window defined by replica.lag.time.max.ms. When a follower falls too far behind, it's removed from ISR. When it catches up, it's re-added.

A message is considered committed only when all ISR replicas have written it. With acks=all and min.insync.replicas=2, a write is acknowledged only after 2 ISR replicas confirm it — guaranteeing durability even if a broker fails immediately after the ack.

If the leader fails, Kafka elects a new leader from ISR — ensuring no committed data is lost. unclean.leader.election.enable=false prevents electing out-of-sync replicas as leaders, which could cause data loss.

6. What are the three delivery semantics? How do you achieve exactly-once?
INTERMEDIATE›

At-most-once: Commit offset before processing. Messages may be lost on crash (consumer committed but didn't finish processing), but never duplicated. Used for non-critical metrics.

At-least-once: Commit offset after processing. Messages are never lost, but may be reprocessed on crash (consumer processed but didn't commit before crashing). Your processing must be idempotent. Most common choice.

Exactly-once: Requires two components: (1) Idempotent producer (enable.idempotence=true) — broker assigns each message a sequence number and deduplicates on retry. (2) Transactions — producer.initTransactions() + beginTransaction() + commitTransaction() — writes to multiple partitions/topics are atomic. Consumer must use isolation.level=read_committed to only read committed transactional messages.

7. Why does Kafka perform so well? What makes it faster than a database?
INTERMEDIATE›

Kafka's performance comes from four key design decisions:

1. Sequential disk I/O: Kafka only appends to the end of log files. Sequential writes on SSDs can exceed 600MB/s — faster than random reads because the OS can aggressively prefetch sequential data. Traditional databases do random I/O for every operation.

2. OS Page Cache: Kafka doesn't maintain its own cache. It writes to the page cache (OS-managed RAM buffer) and lets the OS handle disk persistence. Data written recently is automatically in RAM, making recent reads essentially free.

3. Zero-copy transfers: When sending data to consumers, Kafka uses the sendfile() Linux syscall, which transfers data directly from the OS page cache to the network socket without copying through application memory. This saves 2 data copies and 2 context switches per message batch.

4. Batching and compression: Messages are batched before sending (multiple records in one network request) and compressed end-to-end. A batch of 1000 JSON messages compressed with LZ4 can be 5–10× smaller than individual messages.

8. What is log compaction? When would you use it?
INTERMEDIATE›

Log compaction (cleanup.policy=compact) is a retention mode where instead of deleting old messages by time, Kafka retains the latest record for each key and removes older records with the same key.

This allows a Kafka topic to act as a durable key-value store. A consumer starting from the beginning of a compacted topic will see the latest value for every key — equivalent to a database snapshot.

Primary use cases: Change Data Capture (CDC) topics — if user #123 updates their email 50 times, compaction keeps only the latest. Database replication topics where downstream systems need current state. KTable state restoration in Kafka Streams. Any scenario where current state matters but full history doesn't.

Note: A record with a null value (tombstone) causes the key to be deleted from the compacted log eventually.

9. What causes consumer group rebalancing, and why is it a problem?
INTERMEDIATE›

A rebalance is triggered when: a consumer joins the group, a consumer leaves (crash, graceful close), a consumer is considered dead (heartbeat timeout or max.poll.interval.ms exceeded), or partition count changes.

The problem with eager rebalancing (default pre-2.4): all partitions are revoked from all consumers simultaneously, then reassigned. During this "stop the world" pause, no consumption happens. For high-lag topics or slow consumers, this can cause visible processing delays.

Solutions: (1) Use CooperativeStickyAssignor (Kafka 2.4+) — only partitions being moved are revoked, others continue processing. (2) Tune max.poll.interval.ms to allow enough processing time. (3) Reduce max.poll.records so each batch processes faster. (4) Use session.timeout.ms tuned to your GC pause profile.

// LEVEL 3

Advanced & Architect-Level Questions

10. How would you design a Kafka-based order processing system that guarantees no order is lost and no order is processed twice?
ADVANCED›

Producer side: Enable enable.idempotence=true and use transactions (transactional.id=order-service-1). Use the Outbox Pattern — write the order to the database and an outbox table in a single DB transaction. Debezium captures the outbox table and publishes to Kafka. This guarantees the event is always published exactly once, even if the service crashes between writing to DB and publishing to Kafka.

Kafka configuration: acks=all, replication.factor=3, min.insync.replicas=2, unclean.leader.election.enable=false. This ensures committed messages survive broker failures.

Consumer side: Use isolation.level=read_committed to avoid reading uncommitted transactional messages. Implement idempotent processing — check if the order was already processed before acting (deduplicate by order_id in the processing DB). Commit offsets only after the order is durably persisted in the target system, and wrap the processing + offset commit in a single DB transaction where possible.

End result: The combination of the transactional outbox (no event loss at publish) + idempotent consumer processing (no duplicate effects) gives you effectively exactly-once semantics across the full pipeline.

11. How do you choose the right number of partitions for a topic?
ADVANCED›

Partition count is the most important tuning decision and is hard to change later. Consider these factors:

Throughput: Estimate target throughput ÷ throughput per partition. A single partition can handle ~10–100MB/s depending on hardware. If you need 1GB/s producer throughput and a partition handles ~100MB/s, you need at least 10 partitions.

Consumer parallelism: Number of partitions = max consumer instances in a group. If you'll ever need 20 parallel consumers, you need at least 20 partitions.

End-to-end latency: More partitions = slightly higher latency (more coordination overhead). For low-latency use cases, don't over-partition.

Ordering requirements: If you need all events for entity X in order, ensure your key hash distributes well across partitions. Hotspots (many events to one key) concentrate load on one partition.

Rule of thumb: Start with max(target_throughput / partition_throughput, target_consumer_count). Multiply by 2-3 for growth headroom. For most topics: 6–24 partitions. Never go below 3 for production, don't go above 100 without a strong reason (each partition costs memory and file handles on the broker).

12. Explain KRaft and why it replaces ZooKeeper.
ADVANCED›

Kafka historically used Apache ZooKeeper as an external coordination service for: storing cluster metadata (topic configs, partition assignments), controller election (one broker acts as cluster controller), and authentication/ACL data.

Problems with ZooKeeper: Two systems to operate and monitor. Metadata propagation took seconds (ZK → Controller → Brokers). Cluster could only support ~200K partitions. Controller failover took 30–60 seconds. ZooKeeper had its own scaling limits.

KRaft (Kafka Raft) replaces ZooKeeper with a Raft consensus algorithm built into Kafka itself. A set of brokers are designated as "controllers" and form a Raft quorum. Metadata is stored in an internal Kafka topic (@metadata) replicated via Raft.

Benefits: Single system to operate. Metadata changes propagate in milliseconds. Support for millions of partitions. Controller failover in milliseconds. Simpler deployment (no ZK cluster to manage). KRaft has been production-ready since Kafka 3.3 and ZooKeeper support is deprecated.

13. How does Kafka Streams handle state and fault tolerance?
ADVANCED›

Kafka Streams maintains local state in RocksDB (an embedded key-value store) on each stream processor instance. This enables O(1) lookups for joins and aggregations without network round-trips to external databases.

Fault tolerance: Every state store is backed by a changelog topic in Kafka. Every write to the state store also writes to the changelog. If a stream processor instance crashes, a new instance reads the changelog topic to restore the state — essentially replaying the state changes rather than the raw events. This makes recovery much faster than reprocessing all events.

Standby replicas: Configure num.standby.replicas=1 to have hot standby state on another instance. When the primary fails, the standby only needs to catch up on delta changes since its last sync — recovery in seconds instead of minutes.

Exactly-once in Streams: Use processing.guarantee=exactly_once_v2 to wrap each poll-process-produce cycle in a transaction, ensuring no event is processed twice even on failures.
