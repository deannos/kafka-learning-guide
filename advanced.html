Exactly-Once Semantics (EOS)
Idempotent Producer + Transactions

Exactly-once delivery requires two things: an idempotent producer (prevents duplicate messages on retry) and transactions (atomically write to multiple partitions/topics at once).

The idempotent producer assigns each message a sequence number. The broker deduplicates by rejecting any message with a sequence number it has already seen for that producer session.

Go — confluent-kafka-go: Transactional producer (exactly-once) 
package main

import (
    "fmt"
    "log"
    "github.com/confluentinc/confluent-kafka-go/v2/kafka"
)

func main() {
    p, err := kafka.NewProducer(&kafka.ConfigMap{
        "bootstrap.servers": "localhost:9092",
        "enable.idempotence": true,        // no duplicate messages on retry
        "transactional.id":  "payment-tx-1", // unique per producer instance
        "acks":              "all",
    })
    if err != nil { log.Fatal(err) }
    defer p.Close()

    p.InitTransactions(nil)
    p.BeginTransaction()

    // All three writes are atomic — either all commit or all abort
    sends := []struct{ topic, key, val string }{
        {"payments",       txID,   paymentJSON},
        {"ledger-entries", txID,   ledgerJSON},
        {"notifications",  userID, notifyJSON},
    }
    for _, s := range sends {
        p.Produce(&kafka.Message{
            TopicPartition: kafka.TopicPartition{
                Topic: &s.topic, Partition: kafka.PartitionAny,
            },
            Key: []byte(s.key), Value: []byte(s.val),
        }, nil)
    }

    if err := p.CommitTransaction(nil); err != nil {
        p.AbortTransaction(nil) // rollback all three
        fmt.Printf("Transaction aborted: %v\n", err)
    }
}

Log Compaction & Retention

🗑️
Time-Based Retention
Default mode. Segments older than log.retention.hours (default 168h = 7 days) are deleted. Good for event logs, activity streams, and any data with a natural expiry.
cleanup.policy=delete
🗜️
Log Compaction
For each key, only the most recent record is retained. Older records with the same key are removed. Use for CDC, event sourcing, and database-like topics where you need the latest value per entity.
cleanup.policy=compact
🔀
Compact + Delete
Both policies active. Keys are compacted but records older than the retention period are also deleted. Useful for CDC topics that don't need infinite history but do need latest-per-key semantics.
compact,delete
💡 Compaction Use Case: Debezium CDC
A user updates their email address 50 times. With time-based retention, you'd store all 50 events. With compaction, only the latest user-123 → {email: "[email protected]"} record is kept. New consumers bootstrapping can read just the latest state of every entity without replaying all history.

Go — kafka-go: Stream processing with 5-min windowed fraud detection 
package main

import (
    "context"; "encoding/json"; "log"
    "sync"; "time"
    "github.com/segmentio/kafka-go"
)

// 5-minute sliding window counter per user
type WindowStore struct {
    mu     sync.Mutex
    events map[string][]time.Time
}

func (ws *WindowStore) Add(key string) int {
    ws.mu.Lock(); defer ws.mu.Unlock()
    cutoff, valid := time.Now().Add(-5*time.Minute), ws.events[key][:0]
    for _, t := range ws.events[key] {
        if t.After(cutoff) { valid = append(valid, t) }
    }
    ws.events[key] = append(valid, time.Now())
    return len(ws.events[key])
}

func main() {
    reader := kafka.NewReader(kafka.ReaderConfig{
        Brokers: []string{"localhost:9092"},
        GroupID: "fraud-detector", Topic: "payments",
    })
    writer := &kafka.Writer{
        Addr: kafka.TCP("localhost:9092"), Topic: "fraud-alerts",
    }
    defer reader.Close(); defer writer.Close()

    store := &WindowStore{events: make(map[string][]time.Time)}
    ctx := context.Background()
    for {
        m, err := reader.FetchMessage(ctx)
        if err != nil { log.Fatal(err) }
        var p struct {
            UserID string  `json:"user_id"`
            Amount float64 `json:"amount"`
        }
        json.Unmarshal(m.Value, &p)
        // Count large txns in 5-min window; alert if > 3
        if p.Amount > 10000 {
            if count := store.Add(p.UserID); count > 3 {
                alert, _ := json.Marshal(map[string]any{
                    "user_id": p.UserID, "count": count, "action": "BLOCK",
                })
                writer.WriteMessages(ctx, kafka.Message{
                    Key: []byte(p.UserID), Value: alert,
                })
            }
        }
        reader.CommitMessages(ctx, m)
    }
}
🌊
KStream
Unbounded stream of independent records. Each record is a separate event. Append-only. Think: clickstream, transactions, sensor readings. Every event matters equally.
📋
KTable
Changelog stream where each record updates the value for a key. Represents current state. Think: user profiles, account balances, inventory. Only the latest value per key matters.
🌐
GlobalKTable
Like KTable but replicated to all stream processor instances. Enables joining against small reference data (country codes, product catalog) without re-partitioning the stream.
⏱️
Windowing
Tumbling (non-overlapping), Hopping (overlapping), Session (activity-based gaps), Sliding (event-triggered). Handle late arrivals with grace periods.

Schema Registry

Schema Registry is a centralized service for managing schemas (Avro, Protobuf, JSON Schema) for Kafka topics. It enforces schema compatibility — preventing producers from breaking downstream consumers with schema changes.

Go — confluent-kafka-go: Avro producer with Schema Registry 
package main

import (
    "github.com/confluentinc/confluent-kafka-go/v2/kafka"
    "github.com/confluentinc/confluent-kafka-go/v2/schemaregistry"
    "github.com/confluentinc/confluent-kafka-go/v2/schemaregistry/serde/avro"
)

// Avro schema — versioned and validated centrally
const orderSchema = `{
  "type": "record", "name": "Order",
  "namespace": "com.example.events",
  "fields": [
    {"name": "order_id",   "type": "string"},
    {"name": "user_id",    "type": "string"},
    {"name": "amount",     "type": "double"},
    {"name": "currency",   "type": "string", "default": "USD"},
    {"name": "created_at", "type": "long"}
  ]
}`

type Order struct {
    OrderID   string  `avro:"order_id"`
    UserID    string  `avro:"user_id"`
    Amount    float64 `avro:"amount"`
    Currency  string  `avro:"currency"`
    CreatedAt int64   `avro:"created_at"`
}

func main() {
    // Serializer automatically registers schema & checks compatibility
    srClient, _ := schemaregistry.NewClient(
        schemaregistry.NewConfig("http://schema-registry:8081"))
    ser, _ := avro.NewSpecificSerializer(srClient, serde.ValueSerde,
        avro.NewSerializerConfig())

    p, _ := kafka.NewProducer(&kafka.ConfigMap{
        "bootstrap.servers": "kafka:9092",
    })
    defer p.Close()

    order := &Order{
        OrderID: "ord-001", UserID: "usr-999",
        Amount: 299.99, Currency: "USD", CreatedAt: 1709000000000,
    }
    // On the wire: 5-byte magic header (schema ID) + Avro binary
    val, _ := ser.Serialize("orders", order)
    p.Produce(&kafka.Message{
        TopicPartition: kafka.TopicPartition{
            Topic: strPtr("orders"), Partition: kafka.PartitionAny,
        },
        Key: []byte("ord-001"), Value: val,
    }, nil)
    p.Flush(5000)
}

Kafka Connect

Kafka Connect is a framework for building scalable, fault-tolerant connectors between Kafka and external systems — databases, cloud storage, search engines, and more. No custom code needed for standard integrations.

📥
Source Connectors
Pull data from external systems into Kafka topics. Examples: Debezium (CDC from PostgreSQL/MySQL), JDBC Source (poll tables), S3 Source (read files), Twitter/REST APIs.
📤
Sink Connectors
Push data from Kafka topics to external systems. Examples: Elasticsearch Sink, S3 Sink, JDBC Sink (write to DB), BigQuery Sink, Snowflake Sink.
🔧
Single Message Transforms
Transform messages in-flight without writing code. Built-in SMTs: ReplaceField, MaskField, ExtractTopic, TimestampConverter, Flatten, Cast, and more.

Partition Strategy & Ordering

⚠ Critical: Ordering in Kafka
Kafka guarantees ordering within a partition, not across partitions. To maintain order for related events (e.g., all events for the same order, user, or account), use the same key — Kafka routes records with the same key to the same partition.
🔑
Key-Based (Default)
hash(key) % numPartitions. Same key → same partition → ordering guaranteed per key. Use for: order events, user activity, financial transactions per account.
🔄
Round-Robin
No key provided. Records distributed evenly across partitions. Maximum throughput but no ordering. Use for: independent events where ordering doesn't matter.
🎯
Custom Partitioner
Implement Partitioner interface for business logic. Examples: route VIP customers to dedicated partitions, geographic routing, or size-based load balancing.
🏷️
Sticky Partitioner
Default since Kafka 2.4 when no key. Fills one partition's batch completely before switching. Improves batching efficiency (lower latency, higher throughput vs pure round-robin).
